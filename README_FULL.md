# Что из себя представляет данная CRM система и как работают автоматизации

Это полноценная CRM-система для управления заказами типографии "ПИРАНЬЯ", построенная на платформе Grist. Система включает в себя автоматизации, валидации, бизнес-логику и защиту от ошибок пользователей.

### Основные таблицы системы:

1. **Employees (Сотрудники)** - справочник сотрудников с ролями доступа
2. **Productions (Производства)** - справочник исполнителей (скрыт от пользователей)
3. **Organizations (Организации)** - справочник юридических лиц
4. **Products (Товары/услуги)** - каталог продукции с ценами и себестоимостью
5. **Clients (Клиенты)** - база клиентов с контактами и историей заказов
6. **Orders (Заказы)** - заказы с автоматическими вычислениями и валидацией
7. **OrderItems (Позиции заказа)** - отдельные позиции в заказах с автоматическим расчетом

### Автоматические вычисления и триггеры:

- **Автоматическое вычисление площади** в позициях заказа на основе ширины и высоты
- **Автоматический расчет суммы позиции** с учетом количества, цены за единицу и коэффициентов
- **Автоматический расчет себестоимости** позиции в зависимости от выбранного производства
- **Суммирование стоимости и себестоимости** по всем позициям в заказе
- **Автоматический расчет долга** по заказу (итого минус внесено)
- **Автоматическая генерация номера заказа** и даты создания при создании записи
- **Автоматическое проставление типа плательщика** (юр.лицо/физ.лицо) в зависимости от наличия организации у клиента
- **Автоматическое проставление приоритетного исполнителя** (производства) при выборе товара
- **Автоматическое проставление статуса и менеджера** в позициях заказа на основе данных заказа
- **Автоматическое обновление статуса заказа** на "Отдать в работу" при полной оплате

### Система валидации и контроля:

- **Проверка заполненности обязательных полей** в заказах (клиент и менеджер)
- **Проверка заполненности обязательных полей** в карточках клиентов (ФИО и хотя бы один контакт)
- **Проверка дубликатов клиентов** по телефону
- **Блокировка изменения статуса заказа** при незаполненных обязательных полях клиента
- **Блокировка ручного изменения вычисляемых полей** (суммы, себестоимости, площади и т.д.) для всех пользователей

### Умный поиск и фильтрация:

- **Умная строка поиска для организаций** на основе ИНН и краткого названия
- **Умная строка поиска для клиентов** на основе всех контактных данных (ФИО, телефон, email, telegram, VK, организация)
- **Автоматическая привязка организаций к клиентам** при импорте (малая часть, остальное руками в процессе поступления заказов)
- **Фильтрация заказов и позиций по клиентам, менеджерам, статусам, исполнителям, категориям**

### Защита данных и безопасность:

- **ACL-права доступа** ограничивают редактирование вычисляемых полей и некоторых других
- **Защита от удаления заказов** всеми пользователями
- **Блокировка изменения номера и даты заказа** после создания
- **Разграничение прав доступа** по ролям пользователей

### Технические особенности:

- **Proxy-колонки** обеспечивают надежное обновление триггерных формул
- **Технические колонки** скрывают сложную логику от пользователей
- **Условное форматирование** визуально подсвечивает важные данные и ошибки в заполненных данных
- **Цветовая индикация статусов и менеджеров** помогает быстро ориентироваться в заказах
- **Колонка метода расчёта** в позициях заказа показывает, как рассчитывается цена (за м², шт., пог.м.)
- **Система триггеров** обеспечивает мгновенное обновление связанных данных

## Что делает скрипт

Скрипт автоматически создает в документе Grist:

1. Таблицу `Companies` (юридические лица) с полями:
   - Elba_ID, Name_Full, Name_Short, INN, KPP, Address_Legal, Address_Post, OKPO, OGRN, Contacts_JSON

2. Таблицу `Contacts` (контактные лица) с полями:
   - Full_Name, Phone, Email, Note, Default_Company (ссылка на Companies), Label_Search (формула поиска)

3. Таблицу `Orders` (заказы) с полями:
   - Order_Number, Date_Created (формула), Date_Deadline, Status, Manager (ссылка на Employees),
   - Contact (ссылка на Contacts), Payer_Type, Payment_Subject, Payment_Bill_Num, Delivery_Info,
   - Client_Note, Subject, Tags, Total_Sum, Total_Paid, Debt (формула), Cost_Total

4. Таблицу `OrderItems` (позиции заказа) с полями:
   - Order (ссылка на Orders), Description, Product (ссылка на Products), Contractor (ссылка на Companies),
   - Width, Height, Quantity, Area (формула), Unit_Price, Total_Price (формула), Unit_Cost, Total_Cost, Production_Note

5. Таблицу `Products` (прайс-лист) с полями:
   - Name, Base_Price, Calc_Method

6. Таблицу `Employees` (сотрудники) с полями:
   - Name, Short_Name

Также скрипт настраивает:
- Связи между таблицами
- Формулы для автоматических вычислений
- Права доступа (ограничение на удаление заказов) - настраиваем руками
- Представления (Views) для фильтрации заказов - настраиваем руками


## Список всех точечных улучшений, которые мы внедрили для стабилизации работы базы данных:

### 1. Защита от «пустоты» (NoneType Error)

Во всех математических формулах добавлена проверка `or 0`. Это предотвращает падение скрипта в ошибку, если менеджер еще не успел заполнить поле или случайно удалил значение.

* **Где:** `Debt`, `Area`, `Total_Price_Items`.
* **Результат:** Больше никаких красных треугольников с ошибкой `unsupported operand type`.

### 2. Автоматическое обнуление платежа

Добавлен триггер на колонку `Total_Paid`. Если менеджер полностью стирает сумму, система принудительно ставит `0` вместо пустоты.

* **Результат:** Корректный расчет долга сразу после создания заказа.

### 3. Гибкий ввод коэффициента (наценки)

Формула теперь принимает значение `Markup_Percent` как строку. Она сама удаляет знаки `%`, пробелы и меняет запятые на точки перед расчетом.

* **Результат:** Менеджер может писать «10», «10%» или «10,5» — система поймет всё.

### 4. Интеллектуальное округление (UP-to-10)

Внедрена функция `math.ceil()` для итоговой стоимости позиции. Сумма всегда округляется **вверх до ближайших 10 рублей**.

* **Результат:** Избавление от «копеечных» счетов (вместо 121 руб. будет 130 руб.).

### 5. Безопасный расчет площади

Формула площади обернута в `try-except`. Даже если в колонки ширины или высоты попадет некорректный текст, площадь просто станет равной `0`, не ломая расчет всей таблицы.

* **Результат:** Стабильная работа при частичном заполнении данных.

### 6. Умный авто-статус (Логика «Оплачено»)

В триггер статуса добавлена проверка типа данных `Debt`. Теперь система надежно переводит заказ в статус **«Отдать в работу»** только при выполнении двух условий:

1. Долг равен или меньше 0.
2. Сумма заказа больше 0 (чтобы не запускать в работу пустые строки).

---

**Итог:** База стала «стрессоустойчивой» к человеческим ошибкам и более удобной для бухгалтерии за счет округления. Все эти правки уже адаптированы под твой Python-скрипт.

### ⚡️ Обновления системы (версия 1.8.1) - КРАТКО

* **«Защита от дурака»**: Все формулы переписаны так, чтобы система не выдавала ошибок, если поля не заполнены или стерты.
* **Умный ввод чисел**: Поля (ширина, высота, наценка) теперь понимают и точку, и запятую. Наценку можно вводить со знаком `%`.
* **Авто-округление**: Стоимость позиции теперь автоматически округляется **вверх до 10 рублей** (удобно для расчетов и бухгалтерии).
* **Контроль оплат**:
* Если стереть сумму в «Оплачено», система сама поставит «0» (исключает ошибки в расчете долга).
* Статус **«Отдать в работу»** включается сам только при полной оплате и корректных данных.


* **Точность 2.0**: Все размеры (ширина/высота) принудительно хранятся в формате «0.00», что исключает путаницу в миллиметрах.

---

**Что это дает:** Скорость работы менеджера выше, ошибок в базе меньше, итоговые суммы в счетах выглядят аккуратнее.

# ERP-МОДУЛЬ: Финансовая панель (Закрытый раздел)

## 📋 Обзор системы

ERP-модуль — это админ-панель для владельцев типографии «ПИРАНЬЯ», предназначенная для:
- Управления операционными расходами
- Расчёта заработной платы менеджеров
- Контроля прибыли и убытков
- Учёта личных изъятий владельцев
- Сбора статистики продаж

**Версия системы:** 3.4.0

**Скрипт развёртывания:**
- `5_grist_closed_section.py` — создание финансовой архитектуры (включая колонки статистики)

---

## 🏗️ Архитектура взаимодействия
```
№0 (MonthlyReports) → период-якорь (итоговый отчёт по суммам из №1)
  ├─ №1 (ExpenseCategories) → категории для фильтрации (данные из №3 + №4 + ExtraCosts)
  │   └─ №2 (ExpenseRecipients) → контрагенты для фильтрации (данные из №3)
  │       └─ №3 (ExpenseTransactions) → точка входа данных из API или ручной ввод
  ├─ №4 (SalaryCalculations) → ЗП
  │   ├─ №5 (SalaryBonuses) → бонусы
  │   └─ №6 (OwnerDrawings) → изъятия
  └─ ExtraCosts → разовые расходы
```
### 1. Схема потока данных (Расчёт сумм)
*Откуда берутся цифры и как они считаются для отчёта*

```
└────────────────────┬────────────────────────────────────┘
                     ▲

┌────────────────────┴────────────────────────────────────┐
│  №1 ExpenseCategories (Категории расходов)              │

└─────┬────────────────────┬──────────────────┬───────────┘
      │                    │                  │
      ▼                    ▼                  ▼
┌─────────────┐    ┌─────────────────┐  ┌──────────────┐
│ №3          │    │ №4              │  │ ExtraCosts   │

```

## 📊 Структура таблиц ERP-модуля

### Подготовительная таблица: **ExtraCosts** (Разовые расходы)

**Назначение:** Учёт разовых расходов, связанных с заказами (фрилансеры, монтаж, доставка).

| Колонка | Тип | Описание |
|---------|-----|----------|
| Кому | Text | Исполнитель (монтажник, дизайнер) |
| Категория | Choice | [Монтаж, Доставка, Замер, Дизайн, Расходники, Другое] |

> 📌 Эти расходы **не входят в ExpenseTransactions** (из Эльбы), а учитываются отдельно.
> 📌 Сумма попадает в категорию «Разовые по заказам» → №1 → №0.

#### 🔧 Триггер: Нормализатор телефона (такой же как в CRM)

При вводе номера телефона система автоматически:
1. Удаляет все символы кроме цифр
2. Если номер начинается с `8` и содержит 11 цифр — заменяет `8` на `7`
3. Если номер начинается с `9` и содержит 10 цифр — добавляет `7` в начало
4. Возвращает номер в формате `7XXXXXXXXXX`

### №0. Таблица: **MonthlyReports** (Периодический отчёт)

**Назначение:** Основной периодический отчёт — точка привязки всех данных.

| Колонка | Тип | Описание |
|---------|-----|----------|
| Period_start | Date (Trigger) | Первый день месяца (2026-01-01) |
| Period | Text (Trigger) | «Январь 2026» — для выбора пользователем |
| Period_end | Date (Trigger) | Последний день месяца (2026-01-31) |
| Status | Choice | [Открыт, Закрыт] |
| Threshold_1 | Numeric | План №1 (прибыль) |
| Threshold_2 | Numeric | План №2 (прибыль) |
| Bonus_1 | Numeric | Бонус за план №1 (20000 ₽) |
| Bonus_2 | Numeric | Бонус за план №2 (10000 ₽) |
| Total_Income | Numeric (Formula) | **Доход** — сумма заказов за период |
| Operating_Expenses | Numeric (Formula) | **Расход** — SUM(№1.Total_By_Category) |
| Net_Profit | Numeric (Formula) | **Прибыль** = Доход − Расход |

#### 🔧 Автоматизации

**Period_start (авто-дата при создании):**
- При создании записи автоматически устанавливается первый день текущего месяца
- Время обнуляется (00:00:00)
- При ручном изменении сохраняется введённое значение

**Period (красивое название):**
- Автоматически генерирует название периода в формате «Месяц Год» (например, «January 2026»)
- Используется для выбора пользователем в интерфейсе

**Period_end (конец месяца):**
- Автоматически вычисляет последний день месяца на основе Period_start
- Например: для января 2026 → 31 января 2026

**Total_Income (Доход) — логика расчёта:**
- **Открытый месяц:** считаются все заказы по дате создания (`Date_Created`) за период, кроме статуса «ОТБОЙ»
- **Закрытый месяц:** считаются заказы по дате выдачи (`Date_Deadline`) за период, только со статусами «ОК» и «ОК - отдать»
- Суммируется стоимость всех подходящих заказов (`Total_Order_Amount`)

#### 🔧 Автоматизации

**Total_By_Category — логика расчёта по категориям:**

1. **Зарплатный фонд:** суммирует `Total_Salary` всех менеджеров из SalaryCalculations за период, исключая владельцев (роль ADMIN)

2. **Разовые по заказам:** суммирует стоимость из ExtraCosts за период:
   - Для закрытого месяца: только заказы со статусами «ОК» / «ОК - отдать»
   - Для открытого месяца: все заказы

3. **Обычные категории (Офис, Налоги и др.):** суммирует транзакции из ExpenseTransactions по выбранной категории и периоду

### №4. Таблица: **SalaryCalculations** (Расчёт ЗП)

**Назначение:** Расчёт заработной платы менеджеров и исполнителей.

| Колонка | Тип | Описание |
|---------|-----|----------|
| Employee | Ref: Employees | Сотрудник |
| Period | Ref: MonthlyReports | Период |
| Base_Salary | Numeric (Trigger) | Оклад (из Employees) |
| Commission_Executor | Numeric (Formula) | Премия исполнителю 10% |
| Commission_Margin_10 | Numeric (Formula) | Премия по марже 10% (только Иван Орлов) |
| Commission_Margin_30 | Numeric (Formula) | Премия по марже 30% (только Иван Орлов) |
| KPI_Bonus | Numeric (Formula) | Премия KPI |
| One_Time_Bonuses | Numeric (Formula) | Выплаты / Штрафы |
| Own_Orders | Numeric (Formula) | Свои заказы |
| Drawings | Numeric | Аванс (ручной ввод) |
| Total_Salary | Numeric (Formula) | Итого ЗП |
| Remaining | Numeric (Formula) | Остаток ($Total_Salary - $Drawings) |

#### 🔧 Автоматизации расчёта ЗП

**Base_Salary (оклад):** автоматически копирует значение оклада из карточки сотрудника (Employees.Base_Salary)

**Commission_Executor (премия исполнителю 10%):**
- Находит все заказы периода, где сотрудник указан как исполнитель в позициях заказа (Production.Name = Employee.Name)
- Для закрытого месяца: только заказы со статусами «ОК» / «ОК - отдать» по дате выдачи
- Для открытого месяца: все заказы по дате создания, кроме «ОТБОЙ»
- Суммирует 10% от стоимости позиций заказа

**Commission_Margin_10 (премия по марже 10% — только Иван Орлов):**
- Рассчитывается только для сотрудника с email `89045570154ivan@gmail.com`
- Находит заказы менеджера за период с маржой > 30%
- Суммирует 10% от стоимости этих заказов

**Commission_Margin_30 (премия по марже 30% — только Иван Орлов):**
- Рассчитывается только для сотрудника с email `89045570154ivan@gmail.com`
- Находит заказы менеджера за период с маржой ≤ 30%
- Суммирует 30% от маржи этих заказов (Доход − Себестоимость)

**KPI_Bonus (премия KPI):**
- Не рассчитывается для Ивана Орлова и владельцев (роль ADMIN)
- Если доход периода (Total_Income) >= План №1 → +20 000 ₽
- Если доход периода >= План №2 → +10 000 ₽
- Обе премии суммируются

**One_Time_Bonuses (бонусы):** суммирует все записи из SalaryBonuses для сотрудника за период

**Own_Orders (свои заказы):**
- Находит заказы, где в карточке клиента указана организация "ПИРАНЬЯ" (проверка по ИНН 7811180808)
- Учитываются только заказы, где менеджером указан текущий сотрудник
- Суммирует стоимость этих заказов за период
- Для закрытого месяца: только статусы «ОК» / «ОК - отдать»

**Total_Salary (итого ЗП):**
- **Для менеджеров (EDITOR):** Оклад + все премии + бонусы − Свои заказы
- **Для владельцев (ADMIN):** 50% от чистой прибыли периода − сумма изъятий из OwnerDrawings
- **Для остальных:** Оклад + премии + бонусы − Свои заказы − Аванс

**Remaining (остаток):** Итого ЗП − Аванс (показывает сумму к выплате)

## 📈 Статистика менеджеров (SalaryCalculations)

### Дополнительные колонки статистики

| Колонка | Тип | Описание |
|---------|-----|----------|
| Avg_Margin_Percent | Numeric (Formula) | Средняя маржа в % |
| Avg_Check | Numeric (Formula) | Средний чек (int) |
| Order_Count | Numeric (Formula) | Количество заказов |

#### 🔧 Автоматизации статистики

**Avg_Margin_Percent (средняя маржа %):**
- Находит все заказы менеджера за период
- Для закрытого месяца: только статусы «ОК» / «ОК - отдать»
- Для открытого месяца: все заказы, кроме «ОТБОЙ»
- Вычисляет среднее значение маржи в процентах

**Avg_Check (средний чек):**
- Находит все заказы менеджера за период (с учётом статусов)
- Вычисляет среднюю стоимость заказа (округляется до целого)

**Order_Count (количество заказов):**
- Подсчитывает количество заказов менеджера за период
- Для закрытого месяца: только статусы «ОК» / «ОК - отдать»
- Для открытого месяца: все заказы, кроме «ОТБОЙ»

## 🔐 Права доступа

### Менеджеры
- **Не видят** закрытый раздел (таблицы №0-№6)
- Работают только с таблицами заказов, клиентов, прайса

### Администраторы (владельцы)
- **Видят** закрытый раздел
- **Могут редактировать** только колонки без формул и триггеров:
  - Threshold_1, Threshold_2, Bonus_1, Bonus_2 (MonthlyReports)
  - Base_Salary (Employees)
  - Drawings (SalaryCalculations)
  - Amount, Comment (SalaryBonuses)
  - Amount, Purpose (OwnerDrawings)
  - Все поля ручного ввода в ExpenseTransactions
